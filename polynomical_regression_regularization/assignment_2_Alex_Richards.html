Fit a dataset following a polynomial curve using progressively more complex (higher order) models¶

In this section, we will be fitting linear (1st order), 2nd order, and 4th order regression models to a non-linear dataset to illustrate what polynomial regression is.

Initialize learning rates, data, labels, and weights; define cost, cost derivative, and weight updating functions¶

data = pd.read_csv('ex2data1.csv')

# We will be using the matrix form of gradient descent this time, so we must include a
# column of 1's that will multiply with our bias weights when computing `np.dot(X, W)`
X = np.stack([
        np.ones((len(data))),
        data.iloc[:, 0]],
        axis=1)
y = np.asmatrix(data.iloc[:, -1]).reshape(len(data), -1)

learning_rates = [1, 0.75, 0.5, 0.1]

W_init = np.asmatrix(np.array([0., 0.]).reshape(2, -1))
W = np.copy(W_init)

J = lambda X, y: (1/len(data)) * np.sum(np.dot((np.dot(X, W) - y).T, np.dot(X, W) - y))
dJ_dW = lambda X, y: (2/len(data)) * (np.dot(np.dot(X.T, X), W) - np.dot(X.T, y))

def update_weights(W, alpha):
    W -= alpha*dJ_dW(X, y)
    return W

def train_model(X, y, learning_rates, suppress_results=False, test_split=None):
    global W, W_init
    
    if not suppress_results:
        if test_split:
            rmse = (sqrt(float(J(X, y))), sqrt(float(J(test_split[0], test_split[1]))))
            print('\nTraining RMSE: {}\nTesting RMSE: {}'.format(rmse[0], rmse[1]))
        else:
            print('\nInitial RMSE: {}'.format(sqrt(float(J(X, y)))))

    results = []
    for alpha in learning_rates:
        if not suppress_results:
            print('Performing gradient descent with a learning rate of {}'.format(alpha))
    
        W = np.copy(W_init)
                              
        new_cost, old_cost, start, count = float(J(X, y)), float('inf'), time(), 0
        beginning = start
        
        while new_cost < old_cost and abs(old_cost - new_cost) > 1e-10:
            old_cost = float(new_cost)
            W = update_weights(W, alpha)
            new_cost = float(J(X, y))
            count += 1
        end = time()
        
        if test_split:
            rmse = (sqrt(float(J(X, y))), sqrt(float(J(test_split[0], test_split[1]))))
            print('\nTraining RMSE: {}\nTesting RMSE: {}\nachieved after {time:.2f} seconds and {count} iterations\n'.format( 
                    rmse[0],
                    rmse[1],
                    time=end-beginning,
                    count=count))
        else:
            rmse = sqrt(float(J(X, y)))
            print('\tFinal RMSE: {} achieved after {time:.2f} seconds and {count} iterations\n'.format( 
                    rmse, 
                    time=end-beginning,
                    count=count))
              
        results.append([rmse, alpha, W])
    
    if not suppress_results:
        if test_split:
            best = min(results, key=lambda x: x[0][1])
        else:
            best = min(results, key=lambda x: x[0])
        print('The best model has:\n\tRMSE = {}\n\tlearning rate = {}\n\tW = {}'.format(best[0], best[1], '\n\t    '.join([str(float(r)) for r in best[2]]))) 
    return results

Apply regularization to prevent overfitting in the higher order polynomial regression model¶

If you look at the loss of the best performing models above, you can see that the 16th order model had a RMSE of 0.26 while the 2nd order model had a RMSE of 0.29 -- so the higher order model is clearly better at prediction, right?

If we were to split our dataset into train and test sets, we'd likely see lower evaulation scores using the 16th order model, relative to the 2nd order model, on the test set despite the higher order model having a better evaulation score on the training set. This is because the superfluous parameters of our model are being trained to utilize the noise of the training set to minimize the loss function.

In what follows, we will use ridge (L2 penalty) and lasso (L1 penalty) regression to penalize the model for learning disparate parameter weight values that correlate with overfitting.

Redefine cost and cost derivative functions to include the L2 regularization term; initialize lambda values¶

We include an additional term in the cost function and its derivative which scales with the magnituded of the parameter weights. This forces the model to prefer a more even parameter weight distribution instead of favoring particular features which better minimize the loss (a.k.a. learning the noise of the dataset).

Specifically, the L2 penalty adds the $\lambda$ scaled sum of squared parameter values to the cost function.

In [15]:
J = lambda X, y: (1/len(data)) * (np.dot((np.dot(X, W) - y).T, np.dot(X, W) - y) \
        + np.multiply(l, np.dot(W.T, W)))
dJ_dW = lambda X, y: (2/len(data)) * (np.dot(np.dot(X.T, X), W) - np.dot(X.T, y) \
        + np.multiply(l, np.dot(np.eye(W.shape[0]), W)))
lambdas = [0.01, 0.05, 0.1, 0.25, 0.5, 0.75, 1]

Fit the data using the scikit-learn Ridge model¶

We will now use scikit-learn's Ridge model for an out-of-the-box regularization-based solution. But first, we will introduce the practice of train/test splitting.

Create 70/30 train/test split on the data and labels¶

It is a standard machine learning practice to partition your dataset into train and test splits. The training set is used to fit your model(s) and the test set is used to evaulate the generalization error of the trained models.

The purpose of holding out a test set is as follows. Training a model using the full dataset will almost certainly have a lower accuracy in a real-world setting as the real-world will almost certainly have different data than what was observed during training. Generally, we do not train models so that they can perform optimally on the data we currently have...we train models to perform as well as possible for unseen data. By holding out a test set, we can simulate real-world data and get an evaluation that is a much closer estimate of model accuracy in a real-world deployment.

Also commonly used is the train/validation/test split (typically 60/20/20). This technique involves fitting models to the training data, estimating their prediction error using the validation data (we repeatedly re-train and evaluate while tuning model hyperparameters), and then using the test set only at the very end to get a generalization error for the final model.

If the size of your data is small, then maybe it would be best to use the simpler train/test split -- otherwise the train/validation/test split should be preferred.
